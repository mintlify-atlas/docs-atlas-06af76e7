Ollama enables you to run large language models locally on your own hardware, providing complete privacy and offline capability for threat modeling with STRIDE GPT.

## What is Ollama?

Ollama is an open-source tool that makes it easy to run large language models on your local machine:

<Card>
  **Key Benefits**:
  
  - ✅ **Complete Privacy**: No data sent to external APIs
  - ✅ **Offline Operation**: Works without internet connection
  - ✅ **No API Costs**: Free after initial hardware investment
  - ✅ **Model Library**: Access to 100+ open-source models
  - ✅ **Full Control**: Choose exactly which models to use
  - ✅ **Data Sovereignty**: Sensitive application data stays local
  
  **Trade-offs**:
  
  - ❌ Requires powerful hardware (GPU recommended)
  - ❌ Slower inference than cloud APIs
  - ❌ You manage model downloads and updates
  - ❌ Variable quality depending on model choice
</Card>

## System Requirements

### Minimum Requirements

<Tabs>
  <Tab title="CPU Only">
    **Warning**: CPU inference is very slow
    
    - 16 GB RAM minimum
    - 64 GB RAM recommended for 13B+ models
    - Recent multi-core processor
    - 50+ GB disk space for models
    
    **Performance**: 3-10 tokens/second
    **Viable for**: Small models (7B parameters or less)
  </Tab>
  
  <Tab title="With GPU (Recommended)">
    **NVIDIA GPU** (CUDA support):
    - 8 GB VRAM: 7B-13B parameter models
    - 16 GB VRAM: 13B-34B parameter models
    - 24 GB VRAM: 34B-70B parameter models
    - 48+ GB VRAM: 70B+ parameter models
    
    **Apple Silicon** (M1/M2/M3):
    - 16 GB unified memory: 7B-13B models
    - 32 GB unified memory: 13B-34B models
    - 64+ GB unified memory: 34B-70B models
    
    **Performance**: 20-100+ tokens/second
    **Viable for**: Most threat modeling use cases
  </Tab>
  
  <Tab title="AMD GPU">
    **ROCm support** (AMD GPUs):
    - Linux only (official support)
    - 8+ GB VRAM recommended
    - Check Ollama docs for supported cards
    
    **Performance**: Similar to NVIDIA but less optimized
  </Tab>
</Tabs>

<Warning>
**Hardware Investment**: Quality threat modeling requires at least a 13B parameter model, which needs 16+ GB VRAM (GPU) or 32+ GB RAM (CPU). Budget accordingly.
</Warning>

## Installation

<Steps>
  <Step title="Install Ollama">
    <Tabs>
      <Tab title="macOS">
        Download from [ollama.com](https://ollama.com):
        
        ```bash
        # Or via Homebrew
        brew install ollama
        ```
        
        Start Ollama:
        ```bash
        ollama serve
        ```
      </Tab>
      
      <Tab title="Linux">
        Install via curl:
        
        ```bash
        curl -fsSL https://ollama.com/install.sh | sh
        ```
        
        Start Ollama service:
        ```bash
        systemctl start ollama
        ```
      </Tab>
      
      <Tab title="Windows">
        Download installer from [ollama.com](https://ollama.com) or use WSL2:
        
        ```bash
        # In WSL2
        curl -fsSL https://ollama.com/install.sh | sh
        ollama serve
        ```
      </Tab>
      
      <Tab title="Docker">
        Run Ollama in container:
        
        ```bash
        docker run -d -v ollama:/root/.ollama \
          -p 11434:11434 --name ollama ollama/ollama
        ```
        
        With GPU:
        ```bash
        docker run -d --gpus=all -v ollama:/root/.ollama \
          -p 11434:11434 --name ollama ollama/ollama
        ```
      </Tab>
    </Tabs>
  </Step>
  
  <Step title="Pull Models">
    Download models for threat modeling:
    
    ```bash
    # Recommended models for threat modeling
    ollama pull llama3.3:70b      # Best quality (40 GB)
    ollama pull llama3.1:13b      # Balanced (7.4 GB)
    ollama pull qwen2.5:32b       # Good alternative (19 GB)
    ollama pull deepseek-r1:70b   # Reasoning model (43 GB)
    
    # Faster/smaller options
    ollama pull llama3.1:8b       # Fast (4.7 GB)
    ollama pull phi3:14b          # Efficient (7.9 GB)
    ```
    
    <Info>
    Model sizes shown are approximate. Actual disk usage varies by quantization.
    </Info>
  </Step>
  
  <Step title="Verify Installation">
    Test Ollama is running:
    
    ```bash
    curl http://localhost:11434/api/tags
    ```
    
    Should return list of installed models.
  </Step>
  
  <Step title="Configure STRIDE GPT">
    In STRIDE GPT sidebar:
    
    1. Select **"Ollama"** from model provider dropdown
    2. Enter endpoint: `http://localhost:11434` (default)
    3. Select model from dropdown (auto-discovered)
    4. Adjust timeout if needed (default: 60 seconds)
  </Step>
</Steps>

## Configuration

### Endpoint Configuration

**Default**: `http://localhost:11434`

<Tabs>
  <Tab title="Local Installation">
    Standard local Ollama:
    ```
    http://localhost:11434
    ```
  </Tab>
  
  <Tab title="Remote Server">
    Ollama on another machine:
    ```
    http://192.168.1.100:11434
    http://ollama.internal:11434
    ```
    
    <Warning>
    Ensure firewall allows port 11434.
    </Warning>
  </Tab>
  
  <Tab title="Docker">
    Ollama in Docker container:
    ```
    http://localhost:11434
    ```
    
    Or with custom port mapping:
    ```
    http://localhost:8080
    ```
  </Tab>
  
  <Tab title="Environment Variable">
    Set in `.env`:
    ```
    OLLAMA_ENDPOINT=http://localhost:11434
    ```
  </Tab>
</Tabs>

### Timeout Configuration

**Default**: 60 seconds

<Note>
Larger models or CPU inference may require longer timeouts. STRIDE GPT allows 60-600 second timeouts.
</Note>

Adjust in sidebar:
- **60s**: Fast models (8B) with GPU
- **120s**: Medium models (13B-32B) with GPU
- **300s**: Large models (70B) with GPU
- **600s**: CPU inference or slow hardware

### Model Discovery

STRIDE GPT automatically discovers Ollama models:

```python
# STRIDE GPT queries Ollama API
GET http://localhost:11434/api/tags

# Response includes installed models:
{
  "models": [
    {"name": "llama3.3:70b", ...},
    {"name": "llama3.1:13b", ...}
  ]
}
```

If no models appear:
1. Check Ollama is running: `curl http://localhost:11434/api/tags`
2. Verify models installed: `ollama list`
3. Check firewall/network settings

    ```bash

    # See Ollama docs for specific instructions
    ```
  </Step>
</Steps>

### Memory Management

<Tabs>
  <Tab title="Reduce VRAM Usage">
    Use quantized models:
    ```bash
    # 4-bit quantization (smaller, faster, slight quality loss)
    ollama pull llama3.3:70b-q4_0
    
    # 8-bit quantization (balanced)
    ollama pull llama3.3:70b-q8_0
    
    # Full precision (largest, highest quality)
    ollama pull llama3.3:70b
    ```
  </Tab>
  
  <Tab title="Offload to RAM">
    Split between GPU and CPU:
    ```bash
    # Configure in Modelfile
    # Use when VRAM insufficient for full model
    PARAMETER num_gpu 32  # Layers on GPU
    ```
  </Tab>
  
  <Tab title="Free Memory">
    Unload models when not in use:
    ```bash
    # Ollama keeps models in memory
    # They auto-unload after 5 minutes idle
    
    # Or manually stop Ollama
    pkill ollama
    ```
  </Tab>
</Tabs>

### Concurrent Requests

<Note>
Ollama handles one request at a time by default. For STRIDE GPT, this means:
- Generating threat model: Model fully loaded
- Generating attack tree: Same model, quick start
- Multiple tabs: Sequential processing
</Note>

## Troubleshooting

<Accordion title="Ollama Not Running">
  **Symptom**: "Unable to connect to Ollama"
  
  **Solutions**:
  - Check Ollama is running: `ps aux | grep ollama`
  - Start Ollama: `ollama serve` (or `systemctl start ollama`)
  - Verify endpoint: `curl http://localhost:11434/api/tags`
  - Check firewall: Allow port 11434
</Accordion>

<Accordion title="No Models Found">
  **Symptom**: Dropdown shows "local-model" or empty
  
  **Solutions**:
  - List models: `ollama list`
  - Pull a model: `ollama pull llama3.1:13b`
  - Check Ollama version: `ollama --version`
  - Verify API response: `curl http://localhost:11434/api/tags`
</Accordion>

<Accordion title="Out of Memory Errors">
  **Symptom**: Model fails to load or crashes
  
  **Solutions**:
  - Use smaller model (8B instead of 13B)
  - Use quantized version (q4_0, q5_0)
  - Close other applications
  - Increase swap space (Linux)
  - Check: `nvidia-smi` (GPU memory) or `free -h` (RAM)
</Accordion>

<Accordion title="Very Slow Generation">
  **Symptom**: Threat model takes >5 minutes
  
  **Solutions**:
  - Verify GPU acceleration: `nvidia-smi` shows usage
  - Use smaller model if on CPU
  - Check model quantization (q4 faster than q8)
  - Increase timeout in STRIDE GPT settings
  - Consider cloud API for this use case
</Accordion>

<Accordion title="Poor Quality Results">
  **Symptom**: Threat model incomplete or inaccurate
  
  **Solutions**:
  - Upgrade to 13B+ parameter model
  - Try different model: Qwen, Llama, or DeepSeek
  - Use full precision instead of quantized
  - Provide more detailed application description
  - Consider that Ollama models may not match cloud API quality
</Accordion>

<Accordion title="Timeout Errors">
  **Symptom**: "Request to Ollama timed out"
  
  **Solutions**:
  - Increase timeout in STRIDE GPT sidebar (60s → 300s)
  - Use smaller model
  - Check system resources (CPU/GPU/RAM usage)
  - Ensure model fully loaded (first request slower)
</Accordion>
